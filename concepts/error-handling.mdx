Effect-orpc provides `ORPCTaggedError` to create Effect-native error classes that work seamlessly with both Effect's error handling and oRPC's error system. These errors can be yielded in generators and are automatically converted to oRPC errors.

## Creating tagged errors

Use `ORPCTaggedError` to create error classes with automatic code generation:

```typescript
import { ORPCTaggedError } from "effect-orpc";

// Basic error - code defaults to 'USER_NOT_FOUND' (CONSTANT_CASE of tag)
class UserNotFound extends ORPCTaggedError("UserNotFound") {}

// With explicit code
class NotFound extends ORPCTaggedError("NotFound", { 
  code: "NOT_FOUND" 
}) {}

// With status and message
class ValidationError extends ORPCTaggedError("ValidationError", {
  status: 400,
  message: "Validation failed"
}) {}
```

<Note>
If you don't specify a code, it's automatically generated by converting the tag to CONSTANT_CASE. For example, `UserNotFoundError` becomes `USER_NOT_FOUND_ERROR`.
</Note>

## Error schemas

Add typed data to errors using Standard Schema (like Zod):

```typescript
import { z } from "zod";

class UserNotFoundError extends ORPCTaggedError("UserNotFoundError", {
  schema: z.object({ userId: z.string() })
}) {}

class ForbiddenError extends ORPCTaggedError("ForbiddenError", {
  code: "FORBIDDEN",
  status: 403,
  message: "Access denied",
  schema: z.object({
    reason: z.string(),
    requiredRole: z.string()
  })
}) {}
```

## Using tagged errors in procedures

Define errors in the `.errors()` map to make them available in your procedure:

```typescript
const getUser = effectOs
  .errors({
    UserNotFoundError  // Make error available
  })
  .input(z.object({ id: z.string() }))
  .effect(function* ({ input, errors }) {
    const user = yield* findUser(input.id);
    
    if (!user) {
      // Yield the error directly
      return yield* new UserNotFoundError();
    }
    
    return user;
  });
```

<Warning>
You must add the error class to `.errors()` for it to be recognized by the client as a defined error. Otherwise, it will be treated as an unexpected error.
</Warning>

## Yielding errors

You can yield errors in multiple ways:

```typescript
const procedure = effectOs
  .errors({ UserNotFoundError })
  .effect(function* ({ errors }) {
    // Direct instantiation
    return yield* new UserNotFoundError();
    
    // With data
    return yield* new UserNotFoundError({ 
      data: { userId: "123" } 
    });
    
    // With custom message
    return yield* new UserNotFoundError({ 
      message: "User does not exist" 
    });
    
    // Using Effect.fail
    return yield* Effect.fail(new UserNotFoundError());
    
    // From errors map (uses the key from .errors())
    return yield* errors.UserNotFoundError();
  });
```

## Mixed error maps

Combine traditional oRPC errors with tagged errors:

```typescript
const getUser = effectOs
  .errors({
    // Regular oRPC error
    NOT_FOUND: {
      status: 404,
      message: "User not found",
      data: z.object({ id: z.string() })
    },
    // Effect tagged error
    UserNotFoundError,
    // Custom key for tagged error
    USER_NOT_FOUND: UserNotFoundError
  })
  .effect(function* ({ input, errors }) {
    const user = yield* findUser(input.id);
    
    if (!user) {
      // Use either style
      return yield* errors.NOT_FOUND({ data: { id: input.id } });
      // or
      return yield* new UserNotFoundError();
    }
    
    return user;
  });
```

<Note>
The key in the error map doesn't change the error code. To change the actual error code, set it in the `ORPCTaggedError` options.
</Note>

## Error code generation

The error code is derived from the tag name using CONSTANT_CASE conversion:

```typescript
// Code: "USER_NOT_FOUND_ERROR"
class UserNotFoundError extends ORPCTaggedError("UserNotFoundError") {}

// Code: "VALIDATION_ERROR"
class ValidationError extends ORPCTaggedError("ValidationError") {}

// Code: "ABC_CODE"
class ABCCode extends ORPCTaggedError("ABCCode") {}

// Code: "CUSTOM" (explicit override)
class MyError extends ORPCTaggedError("MyError", { 
  code: "CUSTOM" 
}) {}
```

## Error handling patterns

Handle errors using Effect's error handling operators:

```typescript
const getUser = effectOs.effect(function* ({ input }) {
  const user = yield* findUser(input.id).pipe(
    // Catch specific error tag
    Effect.catchTag("UserNotFoundError", (error) =>
      Effect.succeed({ id: input.id, name: "Guest" })
    )
  );
  
  return user;
});

// Handle multiple error types
const updateUser = effectOs.effect(function* ({ input }) {
  const result = yield* performUpdate(input).pipe(
    Effect.catchTags({
      UserNotFoundError: (e) => Effect.fail(new NotFoundError()),
      ValidationError: (e) => Effect.fail(new BadRequestError()),
      PermissionError: (e) => Effect.fail(new ForbiddenError())
    })
  );
  
  return result;
});
```

## Converting to ORPCError

Tagged errors are automatically converted to `ORPCError` when thrown from procedures:

```typescript
import { toORPCError } from "effect-orpc";

const handler = effectOs.effect(function* () {
  const result = yield* someOperation.pipe(
    Effect.catchTag("UserNotFoundError", (e) =>
      // Manually convert if needed
      Effect.fail(toORPCError(e))
    )
  );
  
  return result;
});
```

The conversion happens automatically in the procedure handler, so you typically don't need to call `toORPCError` manually.

## Error metadata

Access error properties directly:

```typescript
class UserNotFoundError extends ORPCTaggedError("UserNotFoundError", {
  status: 404,
  schema: z.object({ userId: z.string() })
}) {}

const error = new UserNotFoundError({ 
  data: { userId: "123" },
  message: "User not found"
});

// Access properties
error._tag;     // "UserNotFoundError"
error.code;     // "USER_NOT_FOUND_ERROR"
error.status;   // 404
error.message;  // "User not found"
error.data;     // { userId: "123" }
error.defined;  // true

// Convert to ORPC error
const orpcError = error.toORPCError();

// Serialize to JSON
const json = error.toJSON();
```

## Runtime error handling

The Effect runtime automatically handles different error types:

```typescript
const procedure = effectOs.effect(function* () {
  // Tagged error - converted to ORPCError with defined: true
  return yield* new UserNotFoundError();
  
  // Regular ORPCError - passed through
  return yield* Effect.fail(new ORPCError("NOT_FOUND"));
  
  // Other Error - converted to INTERNAL_SERVER_ERROR
  return yield* Effect.fail(new Error("Something broke"));
  
  // Die (unhandled error) - converted to INTERNAL_SERVER_ERROR
  return yield* Effect.die("Fatal error");
});
```

<Tip>
Use tagged errors for expected failure cases and let unexpected errors become `INTERNAL_SERVER_ERROR` automatically.
</Tip>

## Client-side error handling

Tagged errors defined in `.errors()` are recognized by the oRPC client:

```typescript
import { createClient } from "@orpc/client";
import type { Router } from "./server";

const client = createClient<Router>({
  /* config */
});

try {
  const user = await client.users.get({ id: "123" });
} catch (error) {
  if (error.code === "USER_NOT_FOUND_ERROR") {
    // TypeScript knows this is a UserNotFoundError
    console.log("User not found:", error.data.userId);
  }
}
```
